> 在这个背景下值得了解的是：
> 
> 电话只需要双绞线。几十年来企业广泛使用的以太网，同样需要双绞线。现在，这对线缆必须满足更严格的标准才能适用，比如五类线（用于100Mbit）或超五类线（1000Mbit即千兆）——但它本质上就是双绞线，只是比电话线的规格要求更高而已。
> 
> 假设你是个电工，有些活儿要安装电话接口，有些要布设“以太网”（可能百兆以太网就够用），还有些客户明确要求你为千兆网络布线。
> 
> 你当然可以跑去批发商那儿买一盘三类电话线、一盘五类百兆以太网线，再买第三盘超五类千兆网线，然后根据每项任务选用合适的线。只要不出差错，通过在某些工程中使用稍便宜的线缆，或许每年能省下几英镑。
> 
> 或者，你可以直接买一盘超五类线，所有工程都用它。既然是同一盘线，就不会拿错，也无需翻看单据来确认自己在管道里布了正确的线。原本以为是电话线但客户坚称要传数据？没问题，这根本就是同一种线，笑着点头就行。
> 
> 我买现在住的这处房子时，想给这张桌子接千兆以太网，尽管DSL是从另一个房间接入的。我本不愿在墙上打洞，也做好了可能需要打洞的心理准备——结果发现这个房间有电话分线（正如作者所说，我们确实喜欢装电话分机），而且从DSL接入的房间到这儿正好有对双绞线。我打开接线盒一看，哟，居然是超五类线。果然整栋楼都用超五类线布了线，因为就像我说的：何乐而不为呢？这基本上是同样的线缆，何必多备一盘别的？
> 
> 于是我把面板从电话接口换成以太网接口，搞定。

### 20. [GuB-42在“用Rust替代Protobuf”中的新评论](https://news.ycombinator.com/item?id=46732505)
> 我觉得特别讽刺的是，这个标题让人觉得 Rust 带来了 5 倍的性能提升，但实际上它反而拖慢了速度。
> 
> 他们的问题是，软件是用 Rust 写的，但需要用到用 C 语言编写的 libpg_query 库。由于无法直接使用 C 库，他们不得不使用一个 Rust 到 C 的绑定库，这个库为了可移植性采用了 Protobuf。问题在于，这很慢。
> 
> 于是他们做的是：在 LLM 的帮助下，自己编写了一套非可移植但优化得多的 Rust 到 C 绑定。
> 
> 但如果他们当初直接用 C 语言写软件，就完全不需要做任何转换。这意味着，这篇文章其实可以取名为“我们如何降低了使用 Rust 带来的性能损失”。
> 
> 我对 Rust 或 libpg_query 了解不多，但很可能，如果他们彻底去掉转换环节，性能还能更快。不过这很可能涉及大量的适配工作，并且要用到一些 unsafe Rust。编写转换器有很多好处：可移植性、便利性、安全性等等……但它是有代价的。归根结底，我认为这正是为什么计算机硬件那么快，而应用程序却那么慢的一大原因。我们的机器总是在不停地复制、转换、序列化和反序列化各种东西。
> 
> 注：我并不是反对他们的做法，恰恰相反，我一向欣赏那些注重性能的人。他们的做法是合理且有效的，干得漂亮！
